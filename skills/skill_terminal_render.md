# SKILL: 終端渲染引擎 (Terminal Rendering Engine)

> 觸發條件：當討論終端 UI、渲染效能、TUI 設計時

## 渲染管線

```
React Component Tree
    ↓
Ink Reconciler（將 React 元素轉為終端輸出）
    ↓
┌─────────────────────────────────────┐
│ 自訂 ASCII 渲染引擎                  │
│ ├── Int32Array 字元池（預分配）       │
│ ├── Bitmask 編碼樣式元資料           │
│ │   └── bold|italic|color 壓縮成位元  │
│ ├── Diff Patch 優化器               │
│ │   └── 只發送變更的游標移動+字元     │
│ └── 行寬快取（自驅逐 LRU）           │
│     └── stringWidth 呼叫減少 ~50x    │
└─────────────────────────────────────┘
    ↓
ANSI Escape Sequences → Terminal
```

## 技術堆疊

| 技術選擇 | 具體實現 | 設計意圖 |
|----------|---------|---------|
| React + Ink | 終端 UI 框架（~140 元件） | 宣告式 UI = 狀態驅動渲染 |
| Int32Array | 字元池預分配 | 零 GC 壓力 |
| Bitmask 樣式 | bold\|italic\|color 壓縮 | 記憶體效率最大化 |
| Diff Patch | 增量更新 | 消除閃爍 |
| LRU 行寬快取 | stringWidth 結果快取 | CJK/Emoji 計算昂貴 |

## 為什麼用遊戲引擎技術

Token 串流場景（~40-100 tokens/sec）的特殊需求：

1. **每個 token 都觸發 UI 更新** — 傳統「清屏重繪」會閃爍
2. **stringWidth() 是熱路徑** — CJK/Emoji 字元寬度計算特別昂貴
3. **需要即時回應** — 使用者期望打字機效果的流暢度

### 解決方案

| 問題 | 技術 | 效果 |
|------|------|------|
| GC 壓力 | 預分配 Int32Array | 零垃圾回收 |
| 畫面閃爍 | Diff Patch | 只更新變更字元 |
| 寬度計算昂貴 | LRU 行寬快取 | 已計算過的行不重算，減少 ~50x |

## print.ts 的設計挑戰

```
print.ts — ~5,594 行
├── 含 3,167 行 / 12 層嵌套單一函數
└── 負責所有輸出格式化

挑戰：
- 終端寬度動態變化
- Markdown 渲染（程式碼區塊、表格、列表）
- ANSI 跳脫序列正確嵌套
- 多語言字元寬度（CJK 占 2 格）
```

## 設計啟示

對於需要高頻 UI 更新的終端應用：
1. **預分配緩衝區** — 避免每幀分配記憶體
2. **增量更新** — 只發送差異，不清屏重繪
3. **快取昂貴計算** — 字串寬度、ANSI 解析結果
4. **宣告式框架** — React/Ink 讓複雜 TUI 狀態管理更可控
